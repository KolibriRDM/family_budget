# Привязка проекта к Flutter 3.41.2

## Цель

Перевести этот проект на использование локального Flutter SDK из `~/Downloads/flutter_3_41_2` только в рамках данного проекта, без изменения глобальной версии Flutter на компьютере, и привести зависимости проекта в рабочее состояние под этот SDK с минимально необходимыми правками.

## Границы работы

В рамках этой задачи:

- меняем проектные конфиги, которые указывают на Flutter SDK;
- обновляем проектные метаданные, если они зависят от версии Flutter SDK;
- при необходимости корректируем ограничения Dart/Flutter в `pubspec.yaml`;
- пересобираем lockfile и подтягиваем только те версии пакетов, которые нужны для совместимости с Flutter 3.41.2;
- проверяем, что проект как минимум проходит `pub get` и базовую статическую проверку.

Вне рамок этой задачи:

- глобальная смена Flutter SDK в системе;
- установка FVM или перевод всей машины на новый workflow;
- несвязанный рефакторинг приложения;
- массовое обновление зависимостей "до последних" без причины.

## Подход

Рекомендуемый путь для этого проекта:

1. Перепривязать проект к локальному SDK `~/Downloads/flutter_3_41_2`.
2. Синхронизировать project-local Flutter-конфиги и generated-файлы с новым SDK.
3. Проверить совместимость `pubspec.yaml` с Dart SDK, который идёт внутри Flutter 3.41.2.
4. Выполнить обновление зависимостей только настолько, насколько этого требует новый SDK и разрешение зависимостей.
5. Прогнать базовую верификацию и зафиксировать все реальные несовместимости, если они останутся.

## Изменяемые области

Ожидаемые файлы и области изменений:

- `android/local.properties`: новый `flutter.sdk`;
- `ios/Flutter/Generated.xcconfig` и `ios/Flutter/flutter_export_environment.sh`: синхронизация `FLUTTER_ROOT` и связанных generated-параметров;
- `.metadata`: обновление ревизии Flutter-проекта, если это сделает инструмент Flutter;
- `pubspec.yaml`: корректировка `environment.sdk` и, при необходимости, точечных зависимостей;
- `pubspec.lock`: пересборка под новый SDK.

Если в ходе проверки выяснится, что какие-то generated-файлы безопаснее пересоздать инструментом, чем редактировать вручную, приоритет у пересоздания.

## Зависимости и совместимость

Политика обновления зависимостей:

- сначала пробуем разрешить текущий набор пакетов на новом SDK;
- если `pub get` или анализ показывают несовместимость, обновляем только конфликтующие пакеты;
- избегаем обновлений, которые тянут каскадные изменения API без необходимости;
- если пакет требует правок кода приложения, делаем минимальные адаптации, необходимые для прохождения сборки или анализа.

## Риски и как их снижаем

- Локальный абсолютный путь в проектных файлах: это приемлемо, потому что цель именно локально привязать проект к SDK из `Downloads`.
- Generated-файлы могут быть перезаписаны Flutter: поэтому предпочтительнее синхронизировать их через команды Flutter, а не вручную поддерживать несколько источников правды.
- Новый Dart SDK может потребовать поднять ограничение `environment.sdk`: это изменим только при фактической необходимости.
- Некоторые плагины могут конфликтовать с новым SDK или toolchain Android/iOS: будем обновлять их точечно и проверять после каждого шага.

## Верификация

Минимальный набор проверок после изменений:

- запуск `~/Downloads/flutter_3_41_2/bin/flutter --version` в контексте проекта;
- `flutter pub get` через локальный SDK проекта;
- `flutter analyze`;
- при отсутствии блокеров по среде — один из базовых smoke-check сценариев вроде `flutter test` или `flutter build apk --debug`.

Если часть проверок упрётся в внешнюю среду, сеть или локальные инструменты платформы, это будет явно зафиксировано в результате.

## Критерии готовности

Задача считается выполненной, когда:

- проект явно использует `~/Downloads/flutter_3_41_2`;
- зависимости успешно разрешаются под этим SDK;
- не вносятся глобальные изменения в систему;
- нет лишних изменений вне задач совместимости;
- есть понятный отчёт о том, что прошло и что, если осталось, ещё требует внимания.
